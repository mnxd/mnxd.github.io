Git的使用

0.前言

实验室的版本控制从SVN迁移到了git+bitbucket，所以通过一个实际的例子熟悉和走一遍git工作的流程。文章主要分成三个部分来讲，首先是从个人的本地版本控制，再到将本地版本库推送到远程与teammate共享和协作，再到大型项目的workflow，最后会大概的提一下实验室的工作流。

1.本地版本库

1.1git的安装和初始化

git的安装比较简单，直接去官网下载安装即可。安装完成后，在开始菜单里找到“Git”->“Git Bash”，出现如下图所示一个类似命令行窗口的东西，就说明Git安装成功！

因为git是分布式版本控制系统，所以每个机器都必须有自己的标识（用户名和邮箱），所以通过如下语句进行配置。

$ git config --global 参数
$ git config --global user.name "Your Name"
$ git config --global user.email "Email@example.com"

1.2本地库操作

在完成了安装和初始化之后，进入工作文件目录，打开git bash，进行以下步骤：

  • git init 初始化一个git 的版本库

  • git status 检查当前目录下的文件状态（untracked,unstage等）

  • git add filename 将当前文件夹下的filename文件打上标记(git add . 是将当前文件夹下所有文件打标记)，告诉git 下次提交时将该文件进行提交。

  • git commit -m “message” 将打过标记的文件提交

    完成了提交之后，就生成了本地的第一个初始版本，流程的操作如下图所示。

现在就已经把最初的版本提交到版本库了，之后我进行任何版本都只用执行以下三步就可以在版本库里提交了。

$ git status$ git add .$ git commit -m "message"

当然如果当前版本的功能不需要想要回退到之前版本，也可以通过git log 打开版本信息并通过git reset命令回退。

$ git log $ git reset --hard HEAD^(或者是版本号回退到指定版本)、

当然如果有了比较稳定的版本，也可以通过tag标签的方式给某些版本特殊的命名如v1.0等，具体方法在后续补充里，这里先不提。

有了这个初始版本之后，我希望能跟我的队友合作一起来完善这个算法，所以就需要放到一个共享的远程库里

2.远程库和Teamwork

2.1 bitbucket新建库

实验室买了bitbucket，所以这里还是结合bitbuct来讲，当然如果是github，基本操作也是一样的。账号注册略过，先到bitbucket远程新建一个库

2.2连接远程库

新建一个库完成之后，要把本地库推送到远程，就需要建立本地和远程之间的连接，如果用http连接的话每一次请求都会要求输入账号和密码，这样会比较麻烦，所以我们通过ssh公玥的方式，通过下面这个命令就可以在当前电脑上创建一个ssh公玥了。

$ ssh-keygen -t rsa -C your email

如下图所示：

公玥文件可以在图示文件目录下找到，其中的ace是我的电脑用户名。找到ssh文件后打开复制文件内容，在bitbucket的ssh key管理中将复制的内容粘贴添加进去。

添加好后，我们就可以通过git remote这个命令来连接本地和远程库，然后就可以通过git push命令将本地库推送到远程了（注意：请确保本地版本库要有所要推送的版本）

$ git remote add origin  ssh address //ssh address可以在远程库看到，如下图所示
$ git push -u origin master

2.3Teamwork

完成推送后，同组的同学只需要git clone就可以克隆现有仓库到他的电脑，不过如果他也要对项目做出更改并上传到远程库的话，还需要项目所有者在远程库中添加他的账号为成员并给出write及以上的权限，完成之后，小组之内就可以愉快的开始工作了。

$ git clone URL  (URL支持git，ssh，http，https等各种协议)

2.4 gitignore

但是，在查看远程库的时候，发现了一个问题，debug和release是不能上传的（成熟的项目可能有release分支等，但是作为测试性的阶段是不能上传的），我们实验室对这方面有严格要求，无用的文件如debug,release,.log和其他编译过程产生的中间文件时不允许上传到版本库的，所以我就需要赶紧删掉它。

要删除远程库的文件，除了直接在页面上操作以外，还可以通过本地删除再推送的方式如图，但是再我们的代码编写过程中，debug和release是每次调试都会产生的，如果每次都要删除就很麻烦，所以git有一个更好的方法，也就是gitignore文件

我们可以通过touch .ignore创建ignore，这样在本地添加文件的时候就会首先根据ignore文件里的需求先忽略掉不用上传的文件再进行提交，比如图中的文件里就直接忽略了debug 和relsase文件，第三行也忽略了所有.log文件，当然由于之前的提交记录里是包含了这些文件的，所以再每次编辑完ignore文件记得清空暂存区，再重新添加和提交上传。

gitignore文件的语法比较简单，建议参考这篇博客。比如说有些特定情况：要忽略所有的.log文件但是必须保留1.log，就可以通过实现。

2.5Conflict

某天我完成了算法的一部分修改，准备提交时出现以上错误，是因为我准备提交前已经有人提交过更新版本库了，所以B的缓存区和远程库内容不一致，这也就是所谓的冲突了。

解决方案是将远程库拉下来后再提交，成功。可以看出，当两个人修改的地方没有重复的情况下，Git会帮我们自动合并。此时再重新尝试提交就行了

如果对相同的文件做了更改，在git pull 之后，命令行会提示出先冲突的文件，而且当前的分支情况会变成（master| MERGING）表示当前正在进行冲突修改合并中。

打开该文件可以看到，git自动的帮我们把冲突的地方标注了出来，从<<<<<<<<>>>>>>+版本号则是远程库中目前的版本中冲突的地方。

所以这里就需要我们根据实际情况进行修改，完成后再次提交，git就会将当前修改后的版本作为最新的版本进行提交。

2.6冲突解决工具

有一些可视化的冲突解决工具可以便捷的帮助我们解决冲突，有时候发生冲突的地方比较多，所以需要借助工具来查看更方便。

但我个人认为，冲突的数量正常来说应该是不会太多，如果冲突数量过多，那就是任务划分不干净，不同的人直接的工作耦合太高，需要从任务的划分上再做优化。

3.Branch

以上就是组内合作的基本流程，但是实际上的项目进展过程中，一个项目可能会被划分成多个不耦合的部分，如果都在一个master上进行版本控制的话，不同的部分之间一直会出现冲突，虽然由于不耦合每次只用pull一下就自动合并了，但是更好的办法是，用分支来解决。

分支的意思就是如这个图所说，现在我啥都不会，所以我分身一个去学git 一个去学svn ，在学的过程中我们互不冲突，学完之后一合并，就都会了。（图来自廖雪峰官方网站）

我通过一个视频来讲解git 分支的实际操作流程。
在版本回退里，我们已经知道，每次提交，Git都把它们串成一条时间线，这条时间线就是一个分支。截止到目前，只有一条时间线，在Git里，这个分支叫主分支，即master分支。HEAD（*号）严格来说不是指向提交，而是指向master，master才是指向提交的，所以，HEAD指向的就是当前分支。
​ 一开始的时候，master分支是一条线，Git用master指向最新的提交，再用HEAD指向master，就能确定当前分支，以及当前分支的提交点：
​ 所以我们可以通过新建分支的方式，在不同的分支做不同的事情，当我们各自完成各自的部分后再将该部分合并到主干上，这样就减少了冲突也保证了主干文件的正确性，例如如果突然项目中出现了bug，这时如果直接在master上进行bug fix 可能会影响到其他人的工作，而且也会加大bug fix的难度，所以最好的办法就是拉一条bug fix 的分支出来再分支上完成bug fix之后再合并回主干
​ 当然，项目依然是在往前走的，所以很有可能再合并分支时也会出现冲突，这里的冲突解决其实和之前的还是差不多的所以就不讲了。
​ 但是，在你这么做之前，要留意你的工作目录和暂存区里那些还没有被提交的修改，它可能会和你即将检出的分支产生冲突从而阻止 Git 切换到该分支。 最好的方法是，在你切换分支之前，保持好一个干净的状态。 有一些方法可以绕过这个问题（即，保存进度（stashing） 和 修补提交（commit amending））

总的来说，再开发中应该学会多用分支，善用分支，利用分支来解决问题。

4.Workflow

所以我们认为实验室的项目应该由一个管理员账号来管理所有的项目，其他组在做项目时都是通过fork的方式fork一个部分，在完成小组开发之后，向管理员提出pull request请求，再由管理员判断、接收在合并到主干上。

并且所有的项目功能开发都是从master中拉出一条dev分支，再以dev分支为主干来完成的，master分支只用于最稳定版本的发布，此外也还需要专门的release分支来发布release版本等。
